# Orange Skill — Start

바이브코딩 입문자를 **아이디어에서 배포된 웹앱까지** 안내하는 올인원 워크플로다.
사용자는 Claude Pro 플랜을 쓰는 수강생이다 — 사용량이 한정돼 있으니 토큰을 아끼고,
언제든 멈췄다 이어갈 수 있게 진행한다.

사용자가 `/orange-start` 뒤에 아이디어를 함께 적었으면 그것을 출발점으로 삼는다.

## 시작 전

1. **모델 확인** — 이 작업은 Sonnet 4.6에서 진행한다. 모델이 Sonnet이 아니면 사용자에게
   `/model sonnet` 실행을 권한다. (Opus는 사용량을 빨리 소진한다.)
2. 사용자에게 전체 흐름을 한 번 보여준다:

   ```
   기획 → Stitch 프로토타입 → 연결 → 구현 → 완성
   ```

   각 단계의 끝은 **안전하게 멈출 수 있는 지점**이라고 미리 알린다.

## 어느 단계인지 먼저 판단한다

현재 작업 폴더를 확인해 진입 단계를 정한다. `ls`로 파일 존재 여부를 보고, `PLAN.md`가
있으면 그 안의 `## 진행 상황` 체크리스트를 읽는다.

| 폴더 상태 | 진입 단계 |
|-----------|-----------|
| `PLAN.md` 없음 | **기획** |
| `PLAN.md` 있음 + `package.json` 없음 | **연결** (Stitch는 그 사이 단계) |
| `package.json` 있음 + 진행 체크리스트 미완 | **구현** |
| 진행 체크리스트 전부 완료 | **완성** (아래 참고) |

이미 진행 중이던 프로젝트라면 어느 단계부터 잇는지 사용자에게 한 줄로 알린다.

## 단계 실행

판단한 단계의 안내 파일을 **읽고 그대로 따른다**. 한 번에 한 파일만 읽는다 (토큰 절약).

- **기획** → `references/phase-plan.md`
- **연결** → `references/phase-connect.md`
- **구현** → `references/phase-build.md`

한 단계를 마치면:

1. `✅ [단계 이름] 완료 — 안전하게 저장됨. 여기서 멈춰도 됩니다.` 를 출력한다.
2. 사용량이 부족하면 지금 멈추고 다음에 `/orange-resume`로 이어가도 된다고 알린다.
3. 사용자가 계속하면, 다음 단계의 안내 파일을 읽고 자연스럽게 잇는다.

**기획 단계의 끝에는 Stitch 프로토타이핑이 있다.** 사용자가 Stitch 작업을 하러 떠나면
돌아올 때까지 기다린다 — 이 시간에는 Claude 사용량이 들지 않는다. 돌아오면 연결 단계로 잇는다.

## 완성

진행 체크리스트가 모두 완료됐으면:

- 라이브 URL을 다시 안내하고, 직접 열어 확인하도록 한다.
- 더 다듬을 점이 있으면 `references/phase-build.md`의 슬라이스 루프로 돌아간다.
- 없으면 축하하고 마친다. 공유 가능한 URL을 마지막에 다시 보여준다.

## 항상 지키는 원칙

- **스코프 준수** — `PLAN.md`에 없는 기능을 임의로 더하지 않는다. 좋은 새 아이디어는
  `PLAN.md`의 '제외' 항목이나 다음 화면으로 적어두고 넘어간다.
- **자동 저장** — 의미 있는 단위마다 git commit 한다. 사용자가 저장을 신경 쓰지 않게 한다.
- **브라우저 단계 자동화** — "브라우저로 ~하세요"처럼 사용자가 직접 해야 하는 브라우저 일
  (Vercel GitHub App 설치, OAuth 승인, Supabase 토큰·대시보드 등)이 나오면, 떠넘기기 전에
  **Claude in Chrome이 연결됐는지 확인하고(없으면 설치·연결 요청), 가능하면 Claude가 크롬을
  직접 제어해 대신한다** — 사용자는 본인만 할 수 있는 것(로그인·2단계 인증)만 한다. 방법은
  `references/browser-steps.md`를 따른다. 연결이 안 되면 기존 수동 안내로 폴백한다. (단,
  Stitch 프로토타이핑은 자동화하지 않는다 — 직접 써보며 배우는 단계라 사용자가 직접 한다.)
- **만들기 기록** — 각 단계 끝에 프로젝트 루트 `MEMORY.md`에 "무엇을 / 왜 / 어떻게 / 막힌 점"을
  남긴다(단계당 한 항목, 그 단계의 커밋에 함께 담는다). 나중에 복기하거나 수업 사례로 쓰는
  교육 자료다 — **결정의 이유와 막혔다 푼 과정을 구체적으로, 풍부하게 적는다.** 길이를 인위적으로
  줄이지 않는다. 형식·규칙은 `references/memory-log.md`를 따른다.
- **단순하게** — 입문자의 MVP다. 추상화·불필요한 예외처리·미래 대비 설정을 미리 만들지 않는다.
  가장 단순하고 명확한 코드를 쓴다.
- **사용자가 직접 확인** — 화면의 최종 판단은 사용자가 브라우저로 본다 (Claude 토큰이 들지 않는다).
  넘기기 전 가벼운 자기 점검(컴파일·체크리스트)은 하되, 무거운 자동 QA는 하지 않는다.
- **질문은 짧게** — 길게 설명하지 말고 핵심만 묻는다. 입문자가 부담을 느끼지 않게 한다.
