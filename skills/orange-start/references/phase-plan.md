# 기획 단계 — 적응형 인터뷰

목표: 사용자의 아이디어를 **2~3번의 짧은 질문 라운드**로 다듬어, 한 페이지짜리 `PLAN.md`와
Stitch용 프롬프트를 만든다. 질문이 앞 답변에서 이어지되 왕복은 최대 3번이다.

## 시작: 아이디어 파악

1. 사용자의 아이디어를 듣는다 (이미 적었으면 그대로 사용). 비어 있으면 "어떤 걸 만들고
   싶어요?" 라고 한 번만 묻는다.
2. **지금은 어떻게 하고 있는지 확인한다.** 아이디어 설명에 이미 나왔으면 그대로 쓰고, 없으면
   묻는다: "이 일을 지금은 어떻게 하고 계세요? 그 방식에서 가장 불편한 점은 뭔가요?"
   (현재 방식이 없는 새 아이디어면 "없음"도 유효한 답이다.)
   - 현재 방식은 **진짜 데이터와 진짜 고통**을 드러낸다 — 화면·데이터·핵심 가치를 여기에 맞추고,
     현재가 단순하면 MVP도 그만큼 단순하게 한다.
3. **참고 앱/사이트를 확인한다.** 아이디어 설명에 이미 나왔으면 그대로 쓰고, 없으면 묻는다:
   "참고하고 싶은 앱이나 사이트가 있나요? 있으면 이름이나 주소를, 그리고 뭐가 마음에 드는지
   (레이아웃·특정 기능·분위기 등) 알려주세요. 없으면 없다고 하셔도 됩니다."
   - 있으면 무엇을 참고할지 메모한다. 잘 모르는 사이트면 `WebFetch`로 한 번 살펴봐도 좋다 (선택).
   - 이 참고는 뒤의 화면 구성과 Stitch 프롬프트에 반영한다.
   - 2·3·4에서 물어볼 게 남았으면 한 번에 같이 묻는다 (왕복 절약).
4. **다 만들면 누구에게 보여줄지 확인한다.** 아이디어 설명에 이미 나왔으면 그대로 쓰고, 없으면
   묻는다: "이 앱을 만들어 인터넷에 올리면 누구에게 가장 먼저 보여줄 거예요? '회원들' 같은
   범주 말고 — 한 명을 이름이나 관계로요." (구체적인 사람이 없으면 "친구 한 명"도 유효하다.)
   - 한 사람이 정해지면 맥락도 가볍게 확인한다: "그 사람이 이걸 언제 쓸까요?"
   - 이 한 사람이 기획 내내 기준이 된다 — 범주는 설계를 모호하게 두지만 한 사람은 화면 위계·
     로그인·문구를 정해 준다. 마무리에서 배포 URL을 보여줄 대상이기도 하다.
5. 아이디어를 8가지 유형 중 하나로 마음속에 분류한다 (사용자에게 말할 필요 없음):
   업무 자동화 · 내부 도구/대시보드 · 웹앱/신청 페이지 · 데이터 정리/분석 ·
   문서/메일/제안서 자동화 · 챗봇/FAQ · 운영 프로세스 개선 · 발표용 MVP/데모.
   → 어떤 유형이든 **Next.js 웹앱 하나로** 표현한다.
6. 아이디어 설명에서 **이미 답이 나온 항목은 메모**해 둔다. 그건 다시 묻지 않는다.

## 인터뷰 규칙

- **AskUserQuestion에는 사용자에게 보여줄 완성된 질문과 선택지만 넣는다.** 이 파일의 안내
  문구는 *너에게 주는 지시*다. '어떤 질문을 고를지', '질문 가이드', '아래에서' 같은 네 작업
  과정을 질문·선택지 텍스트에 절대 넣지 않는다. 질문은 자연스럽고 완전한 한국어 문장으로 쓴다.
- **한 라운드 = AskUserQuestion 한 번.** 질문 3~4개를 묶어서 던진다.
- 각 질문에는 아이디어에서 추론한 **구체적 선택지 2~4개**를 만들어 준다. 입문자가 글을 쓰는
  대신 고르게 한다 ("기타"로 직접 입력도 가능하다).
- **Round 2·3의 질문은 직전 답변에서 생성한다.** 고정 질문지가 아니다.
- 이미 아는 항목은 건너뛴다. 질문 문구는 짧게 — 긴 설명을 붙이지 않는다.
- **라운드 사이마다 지금까지의 `PLAN.md` 초안을 대화창에 보여준다** — 기획이 채워지는 게
  눈에 보이게. 완성된 `PLAN.md` 파일은 인터뷰가 끝난 뒤 한 번 쓴다.

## Round 1 — 무엇 · 누구 · 핵심

사용자에게 아래 질문들을 **한 번의 AskUserQuestion**으로 묻는다. 아이디어 설명에서 이미 답이
나온 질문은 뺀다 (보통 3~4개를 묻게 된다). 각 질문의 선택지는 사용자 아이디어에서 추론해
2~4개씩 만든다 — 사용자는 "기타"로 직접 입력할 수도 있다.

물어볼 질문 — 아래 문장을 쓰되 '이 도구' 자리에 사용자 아이디어의 실제 대상을 넣어 자연스럽게 만든다:

1. 이 도구를 한 줄로 설명하면 무엇인가요?
2. 이 도구를 주로 누가 쓰나요?
3. 사용자가 이걸로 하는 가장 중요한 행동 하나는 무엇인가요?
4. 잘 되면 사용자에게 어떤 좋은 순간이 생기나요?

답을 받으면 `PLAN.md`의 **1·2절(한 줄 소개·핵심 흐름) 초안**을 대화로 보여준다:
"여기까지 이렇게 이해했어요 — 맞나요?"

## Round 2 — 화면 · 데이터 · 앱 이름

Round 1의 핵심 흐름을 바탕으로:

1. **화면 목록을 대화창에 글로 적어 보여준다.** 핵심 흐름을 3~5개 화면으로 쪼개고, 각 화면마다
   **무엇이 보이고 사용자가 무엇을 하는지** 한 줄로 적는다. 화면 이름만 나열하면 사용자가
   판단할 수 없다 — 반드시 내용을 적는다. 예:

   ```
   제안하는 화면 구성:
   ① 신청 폼 — 이름·연락처·신청 내용 입력칸과 '제출' 버튼
   ② 신청 목록 — 들어온 신청을 표로 표시, 한 줄을 누르면 상세로 이동
   ③ 상세 보기 — 신청 하나의 전체 내용 + '처리 완료' 버튼
   ```

   이어서 알린다: **실제 화면 디자인은 다음 Stitch 단계에서 눈으로 보고 고친다 — 지금은
   "어떤 화면이 필요한지"만 정하는 단계다.** 질문은 이 목록을 적은 뒤에 던진다.
2. 그다음 한 번의 AskUserQuestion으로 아래 4가지를 묻는다:
   - **화면 목록** — "위에 적은 화면들로 충분할까요?" → "그대로 좋음 / 화면 추가 / 화면 빼기 / 둘 합치기".
   - **데이터** — "어떤 정보를 저장해야 하나요?" → 추론한 데이터 항목들을 선택지로 (복수 선택).
   - **로그인** — "로그인이 필요한가요?" → "필요 없음(누구나) / 필요함(사람별 데이터) / 관리자만".
   - **앱 이름** — "앱 이름을 정해요." → 컨셉에 맞는 이름 후보 2~3개를 선택지로 ('기타'로 직접 입력 가능).
     이름은 한글이어도 좋다 — 화면과 `PLAN.md` 제목에 그대로 쓴다.

   메일 발송·파일 업로드 같은 외부 연동은 아이디어에서 대체로 드러난다 — 따로 묻지 말고 네가
   판단해 `PLAN.md`에 반영하고 초안을 보여줄 때 확인받는다. (LLM API는 아래에서 따로 판단한다.)

답을 받으면:
- **앱 이름에서 ASCII slug를 만든다** — 소문자·숫자·하이픈만, 양끝 하이픈 없이 (GitHub·Vercel·
  Supabase·URL에 쓰인다). 한글 이름이면 영문/로마자로 옮긴다 (예: "임팩트 스프린트" →
  `impact-sprint`). 만든 slug를 사용자에게 알린다 — "기술 이름은 `impact-sprint`, 레포·주소에 쓰여요."
- `PLAN.md`의 **3·4절(화면·데이터 모델)까지 초안**을 대화로 다시 보여준다.

## Round 3 — 범위 다듬기 (조건부)

다음 중 하나라도 해당하면 Round 3을 진행한다:

- 화면이 **6개를 넘는다.**
- 한 세션(몇 시간) 안에 다 만들기 **벅차 보인다.**
- 핵심 흐름이 **여전히 모호하다.**

진행한다면 한 번의 AskUserQuestion으로:

- "한 번에 다 만들기엔 양이 많아요. **이번 MVP에 꼭 필요한 화면**을 골라주세요." (복수 선택,
  선택지 = 화면 목록.) 고르지 않은 화면은 `PLAN.md`의 '제외 — 다음에'로 보낸다.

해당 사항이 없으면 Round 3을 건너뛴다. 대신 네가 직접 MVP 범위를 좁혀 `PLAN.md`에 적은 뒤
"이렇게 범위를 잡았어요"라고 보여주고 수정받는다.

## LLM API 필요 여부 판단

아이디어와 핵심 흐름을 보고, 이 앱이 **LLM API**(Claude·GPT 등) 호출이 필요한지 판단한다.

- **필요** — 자유로운 글 생성·요약, 대화형 답변, 자연어 이해·분류가 핵심일 때.
  예: AI 챗봇, 문서 자동 요약, 자유 질문 답변, 내용 자동 작성.
- **불필요** — 정해진 데이터를 입력·저장·표시·집계하는 게 핵심일 때.
  예: 신청 폼, 대시보드, 고정 FAQ(정해진 Q&A 목록), 템플릿 채우기식 문서.

판단 결과를 사용자에게 한 줄로 알리고 `PLAN.md`에 적는다. **필요하다고 판단되면** 이렇게 알린다:
- LLM API는 **API 키**가 필요하고 **호출당 비용**이 든다 — 나머지 스택은 무료지만 이것만 유료다.
- Anthropic 또는 OpenAI 콘솔에서 키를 발급받아야 한다. 다음 Stitch 단계 동안 함께 준비하면 좋다.
- 키는 **서버에서만** 쓴다 (브라우저 노출 금지). 구현 단계에서 서버 라우트로 처리한다.

## PLAN.md 작성

인터뷰가 끝나면 프로젝트 루트에 `PLAN.md`를 아래 형식으로 쓴다. 빌드에 필요한 스펙
(화면·데이터·설정)은 빠짐없이, 각 항목은 핵심만 간결하게 쓴다.
진행 체크리스트의 '구현' 항목은 화면 목록에 맞춰 하나씩 만든다. (git 커밋은 다음 '연결'
단계에서 시작하므로 여기서는 파일만 저장하면 된다.)

각 화면의 `상태:`(로딩·비었을 때·오류)는 **사용자에게 묻지 말고 네가 정한다** — 빈 화면과
무반응 버튼은 입문자가 "고장났다"고 느끼는 가장 큰 원인이다. 목록·대시보드는 로딩 표시·0건
안내·오류 메시지를, 입력 폼은 제출 중·성공·실패 피드백을 갖춘다.

`## 디자인` 절은 아래 템플릿에 **기본값이 이미 채워진 채로** 쓴다 — 빈칸이 아니다. 이 기본값이
디자인의 바닥선이다. Stitch 디자인이 있으면 '연결' 단계에서 실제 값으로 다듬고, 없거나 Stitch
결과가 기본값보다 못하면 기본값을 그대로 둔다.

각 화면 `보임`에는 **1순위**(가장 먼저 보여야 할 것)를 정해 둔다 — 화면에서 모든 게 똑같이
중요하면 사용자 눈이 길을 잃는다.

```markdown
# [표시 이름]

기술 이름(slug): `[ascii-slug]`

## 한 줄 소개
[무엇을 / 누구를 위해 — 한 문장]

## 핵심 흐름
1. [사용자 동작]
2. [사용자 동작]
3. [성공의 순간 — 사용자가 얻는 것]

## 참고 앱/사이트
[참고 대상 + 무엇을 참고할지. 없으면 "없음"]

## 설정
- 로그인: [필요 없음 / 필요(사람별 데이터) / 관리자만]
- LLM API: [필요 (Anthropic 또는 OpenAI) / 불필요]
- 외부 연동: [메일 발송·파일 업로드 등 / 없음]

## 디자인
- 강조색: `#2563EB` (차분한 파랑) — 버튼·링크·강조. 보라색 그라데이션은 쓰지 않는다.
- 배경: `#FFFFFF` · 글자: `#171717` · 카드: `#FFFFFF` · 테두리: `#E5E7EB`
- 상태색: 성공 `#16A34A` · 오류 `#DC2626`
- 폰트: Noto Sans KR
- 모서리(radius): 8px · 그림자: 옅게 카드에만 · 간격: 넉넉하게
- 레이아웃 원칙: [화면 유형에 맞게 — 예: "중앙 1단 폼", "상단 통계카드 + 리스트"]

## 화면
1. **[화면 이름]** — `/[경로]`
   - 보임: [1순위(가장 먼저·가장 크게) → 2순위 → 3순위]
   - 동작: [사용자가 하는 것]
   - 데이터: [읽음: 테이블 · 씀: 테이블]
   - 상태: [로딩 / 비었을 때 / 오류 표시 — 폼이면 제출 중·성공·실패]
   - 디자인: [연결 단계에서 Stitch 파일 경로를 채운다 — 지금은 비워둔다]
2. ...

## 데이터 (Supabase 테이블)
- **[테이블명]**
  - `id` uuid — 기본 키
  - `[컬럼]` [타입(text/int/timestamptz/boolean 등)] — [설명]
  - 관계: [예) user_id → users.id. 없으면 "없음"]

## 기술 스택
Next.js (App Router, TypeScript) · Tailwind CSS · shadcn/ui · Supabase · Vercel

## MVP 범위
- 포함: [이번에 만들 것]
- 제외 — 다음에: [나중으로 미룰 것]

## 진행 상황
- [x] 기획 완료
- [ ] Stitch 프로토타입
- [ ] 연결 (GitHub · Vercel · Supabase)
- [ ] 구현: [화면 1 이름]
- [ ] 구현: [화면 2 이름]
- [ ] 배포 확인
```

## 만들기 기록 시작 (MEMORY.md)

`PLAN.md`를 다 쓴 뒤, 프로젝트 루트에 `MEMORY.md`를 새로 만든다 — 이 앱을 어떤 과정으로
만들었는지 남기는 기록이다 (나중에 복기·교육 사례용). 헤더 + 첫 '기획' 항목을 쓴다:
무엇을·누구를 위해 / 왜 이렇게 정했나(현재 방식의 불편, 보여줄 사람 기준) / 범위를 줄였거나
방향을 튼 지점. **짧게, 5~8줄.** 형식은 같은 폴더 `memory-log.md`를 따른다.
(이 파일은 다음 '연결' 단계의 첫 커밋에 PLAN.md와 함께 담긴다.)

## Stitch 프로토타입으로 넘기기

화면을 코드로 만들기 전에, 사용자가 **눈으로 먼저 확인**하게 한다.

1. `PLAN.md`를 바탕으로 **Stitch 프롬프트 하나**를 만든다 — 영어. 막연한 형용사 대신
   **구체적으로** 쓴다. 아래 순서로 담는다:
   - **무엇·누구** — 어떤 웹앱이고 누구를 위한 것인지 (한 줄 소개에서).
   - **화면별로** — `PLAN.md` `## 화면`(MVP 범위)의 **모든 화면**을 빠짐없이 적는다. 화면
     개수에 제한을 두지 않는다 — MVP에 5개 화면이 있으면 5개를 다 쓴다. 화면마다 `Screen 1`,
     `Screen 2`… 로 번호를 붙여 레이아웃과 핵심 UI 요소를 옮긴다 (PLAN.md 화면 절의 '보임/동작').
     화면 수가 많아 프롬프트가 길어지는 건 괜찮다 — 각 화면 설명을 핵심만 간결하게 쓴다.
   - **스타일 앵커** — "modern·clean" 같은 형용사만 쓰지 말고 구체적 기준으로: 참고 앱이
     있으면 "like [참고 앱]", 없으면 색·톤을 직접 지정 ("soft blue accent, white background,
     rounded corners" 식).
   - **마무리** — "Responsive web app. All UI text in Korean."

   완벽을 노리지 않는다 — 좋은 1차 초안이면 충분하다 (나머지는 Stitch에서 다듬는다).

   잘 만든 프롬프트 예시 — 앱 유형별 3가지. 실제 앱에 가장 가까운 것을 본보기로 삼는다.
   (예시는 화면이 1~2개지만, 이는 분량을 줄인 본보기일 뿐이다 — 실제 프롬프트에는 `PLAN.md`의
   화면을 **개수 제한 없이 전부** 담는다. 같은 형식으로 `Screen 3`, `Screen 4`… 를 이어 쓴다.)

   **① 신청·목록형 — '스터디 신청'**

   ```
   Design a responsive web app for "스터디 신청" — an internal tool for
   employees to sign up for company study groups. All UI text in Korean.

   Screen 1 — 신청 폼: a centered form card on a light background. Heading
   "스터디 신청" with a one-line subtitle. Fields: 이름 (text input),
   부서 (select), 관심 스터디 (select). A full-width primary button "신청하기".

   Screen 2 — 신청 현황: an admin page, heading "신청 현황". At the top, a
   search box and a count label "총 N명 신청". Below, a data table —
   columns 이름, 부서, 스터디, 신청일.

   Style: clean and friendly — white background, soft blue accent, rounded
   corners, generous spacing.
   ```

   **② 대시보드형 — '동아리 회비 현황'**

   ```
   Design a responsive web app for "동아리 회비 현황" — a tool for a club
   treasurer to track and share members' membership-fee payments. All UI
   text in Korean.

   Screen 1 — 현황 대시보드: heading "회비 현황". A top row of three stat
   cards: 이번 달 납부율 (%), 납부 완료 인원, 미납 인원. Below, a member
   list — each row shows 이름, 학번, and a status badge (green "납부 완료"
   or red "미납").

   Screen 2 — 납부 기록: a centered form card, heading "회비 납부 기록".
   Fields: 회원 (select), 납부 월 (select), 금액 (number input). A
   full-width primary button "기록하기".

   Style: clean dashboard look — white background, light-gray cards, soft
   shadows, rounded corners. Green = paid, red = unpaid.
   ```

   **③ 챗봇형 — '사내 규정 안내 챗봇'** (단일 화면)

   ```
   Design a responsive web app for "사내 규정 안내 챗봇" — an AI chatbot
   where employees ask questions about company rules and HR policies and
   get answers. All UI text in Korean.

   One main screen — 대화 화면, a clean chat interface like a messaging app.
   - A top header with the title "사내 규정 봇".
   - A scrollable conversation area: bot replies as left-aligned bubbles,
     the user's messages as right-aligned bubbles in an accent color.
   - Empty state (first load): a short welcome message plus three tappable
     example-question chips — "연차 어떻게 써요?", "재택근무 규정은?",
     "경조사 지원이 있나요?".
   - A fixed bottom input bar: a text field "궁금한 점을 물어보세요" and a
     send button.

   Style: clean and friendly, white background, soft blue accent for the
   user bubbles and send button, rounded corners.
   ```

2. 사용자에게 안내한다:
   - https://stitch.withgoogle.com 에 접속해 위 프롬프트를 붙여넣어 생성한다. 참고 앱이
     있으면 그 캡처를 프롬프트와 함께 올리면 더 비슷하게 나온다.
   - 결과가 기획 의도와 맞는지 본다. 부족하면 **후속 프롬프트로 "더 ~하게" 고치거나** Stitch의
     변형(Create Variant) 기능으로 마음에 들 때까지 다듬는다.
   - 다 됐으면 **그 화면을 연 상태에서** Stitch의 **'프로젝트 다운로드'**로 zip 파일을 받는다.
   - 받은 **zip 파일을 이 프로젝트 폴더 안에 그대로 넣어두고**, 돌아와서 알려준다. 압축은
     풀지 않아도 된다 — 다음 '연결' 단계에서 Claude가 풀어 `design/` 폴더로 정리한다.
     (zip을 받기 어려우면 각 화면 스크린샷을 찍어 `design/` 폴더에 넣어도 된다.)

3. 다음을 꼭 알린다:
   - **Stitch 작업은 Claude 사용량을 쓰지 않는다.** 충분히 시간을 들여도 된다.
   - **Stitch 결과가 생각과 다르면, 그건 기획이 어긋났다는 신호다.** `PLAN.md`를 고치고
     프롬프트를 다시 만들어 Stitch를 다시 돌린다.
   - **Stitch 결과가 제네릭하거나 어설프면 억지로 쓰지 않는다.** `## 디자인`의 기본값이 더
     나으면 기본값을 쓴다 — Stitch는 기본값을 끌어올릴 때만 가치가 있다.

`✅ 기획 완료 — PLAN.md 저장됨. 멈춰도 됩니다.` 를 출력한다. 사용자가 Stitch에서 돌아오면
연결 단계(같은 폴더의 `phase-connect.md`)로 잇는다.
