# 구현 단계 — 화면을 하나씩 만든다

목표: `PLAN.md`의 화면을 **한 번에 하나씩** 진짜로 만든다. 화면 하나가 끝날 때마다 저장·배포되고,
사용자가 직접 확인한다.

## 준비

1. `PLAN.md`를 읽는다 — 특히 `## 진행 상황` 체크리스트와 `## 설정`·`## 디자인`·`## 화면`·`## 데이터`.
   각 화면의 `디자인:` 줄(연결 단계에서 박아둔 화면↔파일 매핑)도 확인한다.
2. `design/` 폴더의 Stitch 산출물이 시각 기준이다 (있다면). **각 화면을 만들기 직전에 그 화면의
   `디자인:` 파일을 실제로 열어 읽는다** — 미리 다 읽지 말고 화면별로 그때 읽는다(토큰 절약).
3. 개발 서버를 켠다: `npm run dev` (백그라운드 실행). 사용자에게 http://localhost:3000 을
   열어두라고 안내한다. 코드를 저장하면 화면이 자동으로 새로고침된다.

## 세로 슬라이스 루프

진행 체크리스트에서 **아직 안 끝난 '구현' 항목 하나**를 고른다. 그 화면 하나를 끝까지 만든다:

### 1) 데이터 준비
이 화면이 쓸 Supabase 테이블이 아직 없으면 만든다.
- `PLAN.md` `## 데이터` 모델(컬럼·타입·관계)대로 테이블 SQL(`create table ...`)을 작성해
  `supabase/` 폴더에 `.sql` 파일로 저장한다 (예: `supabase/001_<테이블>.sql`). 사용자에게도 보여준다.
- **CLI로 바로 실행한다** (연결 단계에서 `supabase link`를 해 둔 상태):

  ```bash
  supabase db query --linked --file supabase/001_<테이블>.sql
  ```

  `db query --linked`는 Management API로 실행되어 DB 비밀번호가 필요 없다. 오류가 나면 같은
  폴더의 `troubleshooting.md` 'Supabase 테이블 생성 실패' 항목을 본다.
- **RLS는 항상 켠다 — 예외 없음.** `public` 스키마의 모든 테이블에서 RLS가 꺼져 있으면 프로젝트
  URL만 알면 누구나 read/edit/delete 할 수 있다(Supabase가 이메일로 경고를 보낸다). 테이블 SQL
  마지막에 항상 다음을 붙인다:

  ```sql
  alter table public.<테이블> enable row level security;
  ```

  그 다음 `PLAN.md` `## 설정`의 **로그인** 항목에 따라 정책을 추가한다:
  - **로그인 없음 + 공개 데이터**(공개 게시판 등): 명시적 public 정책을 둔다.
    ```sql
    create policy "anyone can read" on public.<테이블> for select using (true);
    create policy "anyone can insert" on public.<테이블> for insert with check (true);
    -- update/delete는 정책을 두지 않아 anon 키로는 막힌다
    ```
  - **로그인 없음 + 비공개 데이터**(폼 제출·신청 접수·관리자 뷰 등 — 사용자는 자기가 넣은 것만
    보거나 아예 못 봐야 하는 경우): 클라이언트는 RLS로 차단하고 **쓰기·읽기는 서버 라우트**
    (`app/api/<이름>/route.ts`)에서 `SUPABASE_SERVICE_ROLE_KEY`로 처리한다. service_role 키는
    `.env.local`에 `SUPABASE_SERVICE_ROLE_KEY=...`로 두고 **`NEXT_PUBLIC_` 접두사를 절대 붙이지
    않는다**(브라우저 노출 금지). Vercel env에도 똑같이 등록한다.
    ```sql
    -- 정책을 두지 않으면 anon은 전부 차단, service_role은 RLS를 우회한다
    ```
  - **로그인 있음**: Supabase Auth(이메일/비밀번호)를 쓰고, 테이블에 `user_id` 컬럼과
    `auth.uid() = user_id` 정책을 둔다.
    ```sql
    create policy "owner read" on public.<테이블> for select using (auth.uid() = user_id);
    create policy "owner write" on public.<테이블> for insert with check (auth.uid() = user_id);
    create policy "owner update" on public.<테이블> for update using (auth.uid() = user_id);
    create policy "owner delete" on public.<테이블> for delete using (auth.uid() = user_id);
    ```

### 2) 화면 구현

**먼저 이 화면의 Stitch 디자인을 읽는다.** `PLAN.md` 화면 항목의 `디자인:` 파일을 연다 (`디자인:
없음`이면 PLAN.md 글 설명으로 만든다). HTML에서 아래를 그대로 파악한다 — 이게 "Stitch와 다른
결과"를 막는 핵심이다:

- **레이아웃 구조** — 영역 배치와 순서 (헤더 → 통계카드 → 표, 좌우 분할, 1단/2단 등).
- **요소 목록** — 어떤 입력 칸·버튼·표 열·항목·뱃지가 있는지. 빠뜨리거나 더하지 않는다.
- **순서·그룹핑·위계** — 무엇이 위에/크게 오는지. 이 순서를 바꾸지 않는다.
- **실제 문구(라벨)** — 버튼·제목·칼럼명 등 한국어 텍스트를 디자인 그대로 쓴다.

그다음 구현한다:

- `app/` 아래에 해당 화면의 페이지(`app/<경로>/page.tsx`)와 필요한 컴포넌트를 만든다.
- **`components/ui/`의 shadcn/ui 컴포넌트를 우선 쓴다.** 버튼·입력·카드·표·뱃지 등은 직접
  만들지 말고 `Button`·`Input`·`Card`·`Table` 등을 조합한다. 없는 컴포넌트는
  `npx shadcn@latest add <이름>`으로 추가한다. 검증된 컴포넌트가 결과를 깔끔하게 만든다.
- **"그대로 복사하지 않는다"의 뜻**: Stitch의 **HTML 마크업·인라인 스타일·하드코딩 색**을
  붙여넣지 않는다는 뜻이다 (그건 shadcn 컴포넌트 + 토큰으로 다시 짠다). **레이아웃·요소·순서·
  라벨은 디자인을 충실히 따른다** — 이걸 마음대로 바꾸면 "Stitch와 다른 화면"이 된다. 디자인은
  출발점이되, 구조는 지키고 구현 방식만 우리 스택으로 옮기는 것이다.
- **색·모서리는 새로 정하지 않는다.** 디자인 토큰은 연결 단계에서 `globals.css`의 shadcn 변수에
  전부 반영돼 있다 — shadcn 컴포넌트가 그 값을 자동으로 들고 온다. 색이 필요하면 임의 hex 대신
  토큰 클래스(`bg-primary`·`text-foreground`·`border-border`)를 쓴다. 성공·오류 상태색은 shadcn에
  없으니 Tailwind 기본 클래스(`green-*`·`red-*`)를 쓰면 된다.
- **화면의 시각적 1순위를 지킨다.** `PLAN.md` 화면 절 `보임`의 1순위 요소를 화면에서 가장
  크고 먼저 오게 한다 — 모든 요소가 같은 크기·무게면 사용자 눈이 길을 잃는다.
- **`상태:` 명세를 빠짐없이 구현한다.** `PLAN.md` 화면 절의 상태를 전부 만든다 — 목록·대시보드는
  로딩 표시·0건 안내 문구·실패 시 오류 메시지를, 입력 폼은 제출 중·성공·실패 피드백을 둔다.
  빈 화면과 무반응 버튼은 입문자가 "고장났다"고 느끼는 가장 큰 원인이다.
- **스타일은 Tailwind 유틸리티 클래스(`className`)로** 한다. `app/globals.css`는 shadcn와 연결
  단계에서 이미 설정됐으니 구현 중에는 건드리지 않는다 — 특히 `@import` 줄은 절대 손대지
  않는다 (Tailwind v4 파서가 깨진다). 본문 폰트(Noto Sans KR)도 연결 단계에서 이미 설정됐다.
- `lib/supabase.ts`의 `supabase` 클라이언트로 데이터를 읽고 쓴다.
- **URL·식별자는 ASCII만 쓴다.** 한글은 화면 표시용으로만 — URL 경로·slug·DB 조회 키에 한글이
  들어가면 URL 인코딩이 꼬여 조회가 깨진다(Supabase 406 등). 한글 이름에서 slug를 만들 땐
  한글을 남기지 말고 짧은 랜덤 ASCII id를 쓴다 (예: `crypto.randomUUID().slice(0, 8)`).
- **LLM API** — `PLAN.md`가 LLM API 필요라고 하면, LLM 호출은 **서버 라우트**
  (`app/api/<이름>/route.ts`)에서만 한다 — 비밀 키를 브라우저에 노출하지 않는다. 발급받은 키에
  맞는 SDK(`@anthropic-ai/sdk` 또는 `openai`)를 설치하고, 화면은 그 라우트를 `fetch`로 호출한다.

### 3) 자기 점검 — 사용자에게 넘기기 전에

입문자가 오류 화면이나 어설픈 화면을 처음 발견하게 두지 않는다. 넘기기 전에 직접 거른다.

**(1) 컴파일·렌더 점검** — 만든 화면을 직접 쳐 본다 (백그라운드 dev 서버가 떠 있는 상태에서):

```bash
curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/<화면 경로>
```

`200`이면 통과. `500`이면 빌드/렌더 오류다 — 백그라운드 dev 서버 출력에서 오류 메시지를 찾아
**사용자에게 넘기기 전에 고친다** (해결은 아래 '막혔을 때'를 따른다).

**(2) 슬롭 체크리스트** — 아래를 빠르게 자기 검토하고, 걸리는 게 있으면 고친 뒤 넘긴다:

- [ ] (Stitch 디자인이 있으면) 이 화면의 `디자인:` 파일을 실제로 열어 봤고, 레이아웃·요소·순서·
      라벨이 그 디자인과 일치한다 (색·코드만 토큰/shadcn로 대체했다).
- [ ] `상태:` 3종(로딩·비었을 때·오류, 폼이면 제출 피드백)이 전부 구현됐다.
- [ ] 색·모서리는 토큰 클래스(`bg-primary` 등)로 했다 — 임의 hex를 새로 쓰지 않았다.
- [ ] 버튼·입력·카드·표를 shadcn/ui 컴포넌트로 만들었다.
- [ ] 화면의 1순위 요소가 가장 크고 먼저 온다 (모든 게 같은 크기 아님).
- [ ] 보라 그라데이션·과한 그림자·발광·장식 이모지 같은 "AI 티" 장식이 없다.
- [ ] 아이콘+제목+짧은 설명 카드를 3개씩 똑같이 줄세우지 않았다 (가장 흔한 AI 레이아웃).
- [ ] 모든 것을 가운데 정렬하지 않았다 — 본문·목록·표는 왼쪽 정렬이 기본.
- [ ] 제네릭 카피가 없다 ("환영합니다", "올인원 솔루션" 류) — 실제 내용을 쓴다.
- [ ] 본문 글자 대비가 충분하다 (작은 회색 글씨 금지).
- [ ] 좁은 폭(모바일)에서도 레이아웃이 깨지지 않는다.
- [ ] `PLAN.md`에 없는 화면·기능을 임의로 더하지 않았다.
- [ ] 이 화면이 쓰는 모든 `public` 테이블에 RLS가 켜져 있고 정책이 명시돼 있다.
- [ ] `service_role`(또는 `secret`) 키가 `NEXT_PUBLIC_*`로 노출되지 않았다 — 서버 라우트에서만 쓴다.

**(옵션) 미리보기** — 시각까지 확실히 보려면 Claude Code의 Preview 도구로 화면 스크린샷을 한
장 찍어 확인할 수 있다. 토큰을 조금 더 쓰므로 필수는 아니다 — 보통은 위 점검과 다음의 사용자
확인으로 충분하다.

### 4) 사용자 확인
- localhost:3000의 해당 화면을 사용자가 직접 보게 한다: "이 화면 어때요? 생각대로 됐나요?"
- 수정을 요청하면 반영하고 다시 보게 한다. (이 확인은 Claude 사용량을 쓰지 않는다.)

### 5) 저장 + 배포
화면이 동작하면:

```bash
git add -A && git commit -m "구현: <화면 이름>"
git push
```

`git push`하면 Vercel이 자동 배포한다. (배포가 안 되면 같은 폴더 `troubleshooting.md`의
'git push가 자동배포 안 됨' 참고.)
`PLAN.md` 진행 체크리스트에서 이 화면 항목을 `- [x]`로 바꾼다.

`✅ <화면 이름> 완료 — 저장·배포됨. 멈춰도 됩니다.` 를 출력한다.

### 다음 화면으로
체크리스트에 안 끝난 화면이 남았으면 1)로 돌아간다. **한 번에 한 화면씩.**

## 막혔을 때

오류가 나면 추측으로 이것저것 바꾸지 않는다:
1. **오류 메시지를 그대로 읽는다.** 어디서, 무엇이 문제인지 정확히 본다.
2. 같은 폴더의 `troubleshooting.md`를 읽어 흔한 원인과 맞는지 본다.
3. **근본 원인**을 고친다. 증상만 가리지 않는다.
4. 다시 확인한다.

## 마무리

모든 화면 항목이 `- [x]`가 되면:

1. **폴리시 패스** — 앱 전체를 화면 하나씩 빠르게 다시 본다. '자기 점검'의 슬롭 체크리스트를
   기준으로, 화면 사이에 어긋난 점(화면마다 다른 색·간격·버튼 모양, 빠진 상태 처리)을 한 번에
   다듬는다. 큰 개편이 아니라 **일관성 손질**이다 — 한 슬라이스 분량으로 끝낸다.
2. 백그라운드 dev 서버를 멈춘 뒤 `npm run build`로 배포용 빌드가 통과하는지 확인한다
   (`dev`와 `build`를 동시에 돌리지 않는다 — `.next`가 충돌한다). 실패하면 고친다.
3. **이제 배포한다.** 화면을 다 만들었으니 여기서 처음으로 라이브 URL을 확인한다 (연결 단계에선
   빈 화면을 배포하지 않았다). 구현 중 `git push`로 이미 자동 배포돼 있으면 그 프로덕션 URL을
   `vercel ls`(또는 `https://<slug>.vercel.app`)로 확인하고, 아직이면 `vercel --prod`로 배포해
   URL을 받는다. URL을 열어 마지막으로 확인한다.
4. `README.md`에 이 앱이 무엇인지 한 단락 적는다.
5. `MEMORY.md`에 '구현' 항목을 **한 개만** 덧붙인다(append) — 화면마다 쓰지 않고 구현 전체를
   하나로 묶는다. 만든 화면 목록 + **핵심 기술 결정 2~3개**(예: 로그인/RLS 처리 방식, 데이터
   구조 선택, 외부 이미지·API를 어떻게 붙였나)와 **그렇게 정한 이유** + **구현 중 막혔다 푼
   지점**(증상·원인·해결책)을 구체적으로. 헤매다 푼 자리가 가장 값진 기록이니 자세히 적는다.
   형식은 `references/memory-log.md`를 따른다.
6. `PLAN.md`의 `- [ ] 배포 확인`을 `- [x]`로 바꾸고 (`MEMORY.md`·`README.md`와 함께) 커밋·푸시한다.
7. 사용자에게 **공유 가능한 라이브 URL**을 보여주며 축하한다. 기획 때 정한 '보여줄 사람'에게
   지금 바로 보내보라고 권한다. (발표·데모에 바로 쓸 수 있다.)

## 원칙

- **스코프 준수** — `PLAN.md`에 없는 화면·기능을 임의로 더하지 않는다. 새 아이디어는 `PLAN.md`의
  '제외 — 다음에'에 적어두고 넘어간다.
- **작은 슬라이스** — 한 번에 한 화면. 여러 화면을 동시에 건드리지 않는다.
- **단순하게** — 입문자 MVP다. 추상화·불필요한 예외처리를 만들지 않는다. 가장 단순한 코드를 쓴다.
- **사용자가 확인** — 화면의 최종 판단은 사용자가 브라우저로 한다. 그 전에 가벼운 자기 점검
  (컴파일 점검 + 슬롭 체크리스트)으로 거른다. 무거운 자동 QA·서브에이전트는 쓰지 않는다.
