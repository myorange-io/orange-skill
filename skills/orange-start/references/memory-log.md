# 만들기 기록 — MEMORY.md 작성 규칙

프로젝트 루트의 `MEMORY.md`는 **이 앱을 어떤 과정으로 만들었는지** 남기는 기록이다.
나중에 복기하거나, 수업에서 "이렇게 만들어졌다"를 사례로 보여줄 때 쓴다.

세 스킬(orange-start · orange-resume · orange-secure)이 이 한 파일을 공유한다.

## 핵심 원칙

- **풍부하게 남긴다.** 이 기록은 나중에 사람이 읽고 배우는 교육 자료다 — 한두 줄로 요약해
  버리면 정작 배울 점(왜 그렇게 정했는지, 무엇이 막혔고 어떻게 풀었는지)이 사라진다. 단계당
  **충분히 자세한 항목**을 남긴다. 길이를 인위적으로 줄이지 않는다 — 결정·시행착오·배운 점이
  있으면 다 적는다. 항목이 길어지는 건 좋은 일이다. **단, 빈약한 채우기는 금물이다 — 줄 수를
  채우려 뻔한 말을 늘리지 말고, 실제로 일어난 결정과 막힘을 구체적으로 적는다.**
- **append만 한다.** 기존 내용을 **덮어쓰지 않는다.** 늘 파일 끝에 새 항목을 덧붙인다.
  파일이 길어지는 건 정상이다 (그게 기록의 목적이다).
- **결과가 아니라 과정을 남긴다.** "무엇을 만들었다"가 아니라 **무엇을 / 왜 그렇게 정했나 /
  어떻게 했나 / 뭐가 막혔거나 바꿨나 / 무엇을 배웠나**를 적는다. 특히 **막힘과 그 해결**,
  **버린 선택지와 그 이유**가 교육 사례의 핵심이다 — 잘 풀린 것보다 헤맨 자리가 더 값지다.
- **구체적으로 쓴다.** "오류가 나서 고쳤다"가 아니라 **무슨 오류 메시지였고, 원인이 뭐였고,
  무슨 명령/수정으로 풀었는지**를 적는다. 나중에 같은 데서 막힌 사람이 그대로 따라할 수 있게.
- **사람 말로 쓴다.** 입문자가 나중에 읽어도 이해되게, 기존 스킬과 같은 한국어 문체로.
  전문용어가 나오면 한 줄로 풀어 준다.
- **날짜는 셸로 얻는다.** `date +%Y-%m-%d` 결과를 항목 제목에 넣는다.
- **별도 커밋을 만들지 않는다.** 각 단계가 어차피 하는 git commit에 `MEMORY.md`를 함께 담는다.

## 파일이 없으면 — 헤더부터 만든다 (기획 단계에서 1회)

`MEMORY.md`가 없으면 아래 헤더 + 첫 '기획' 항목으로 새로 만든다:

```markdown
# 🍊 만들기 기록 — [앱 이름]

> 이 앱을 어떤 과정으로 만들었는지 남기는 기록입니다.
> 나중에 복기하거나, 다른 사람에게 사례로 보여줄 때 씁니다.

## 한눈에 보기
- 무엇을: [PLAN.md 한 줄 소개]
- 누구를 위해: [주 사용자 / 보여줄 사람 한 명]
- 핵심 흐름: [PLAN.md 핵심 흐름 한 줄 요약]
- 스택: Next.js · Tailwind · shadcn/ui · Supabase · Vercel
- 시작: [YYYY-MM-DD]   · 라이브: 배포 전

---

## 기록

### [YYYY-MM-DD] 기획
- **정한 것**: [핵심 흐름·화면 수·로그인/LLM 여부 등. 두세 줄로 구체적으로]
- **왜**: [그렇게 정한 이유 — 지금 이 일을 어떻게 하고 있고 뭐가 불편한지, 보여줄 사람이
  누구라 무엇이 중요한지. 진짜 동기를 적는다]
- **고민하다 버린 선택지**: [넣을까 했다가 뺀 화면·기능과 그 이유. 없으면 생략]
- **막힌 점 / 바꾼 점**: [범위를 줄였거나 방향을 튼 지점과 그 계기. 없으면 "없음"]
```

이미 `MEMORY.md`가 있으면 **헤더를 다시 만들지 말고** 아래 항목 형식만 파일 끝에 덧붙인다.

## 항목(entry) 형식 — 매 단계 공통

각 단계 끝에 아래 골격으로 항목을 덧붙인다. 모든 줄을 억지로 채울 필요는 없지만,
**"왜"와 "막힌 점 / 바꾼 점"은 거의 항상 적을 게 있다** — 비우지 말고 구체적으로 채운다.

```markdown
### [YYYY-MM-DD] [단계 이름]
- **정한 것 / 한 것**: [이 단계에서 실제로 만들거나 정한 것]
- **왜**: [그 선택의 이유 — 다른 길도 있었는데 왜 이걸로?]
- **어떻게**: [구체적 방법·도구·명령. 나중에 따라할 수 있을 만큼]
- **막힌 점 / 바꾼 점**: [무슨 증상이었고, 원인이 뭐였고, 무슨 명령/수정으로 풀었나.
  방향을 바꿨다면 무엇 때문에. 없으면 "없음"]
- **배운 것 / 다음**: [이 단계에서 알게 된 것, 또는 다음에 이어서 할 것. 없으면 생략]
```

단계 이름은 그 단계에 맞게 쓴다: `기획` · `연결` · `구현` · `이어가기` · `보안 점검`.
배포 URL이 처음 생기면 '한눈에 보기'의 `라이브:` 한 줄을 실제 URL로 갱신한다 (이건 덮어쓰기 예외).

## 단계별 무엇을 적나

| 단계 | 스킬 | 남길 것 |
|------|------|---------|
| 기획 | orange-start / phase-plan | 헤더 생성 + 무엇을·누구를·왜 이 범위로 정했나 + 버린 선택지 |
| 연결 | orange-start / phase-connect | 스택·배포·DB를 **어떻게** 붙였나, 막힌 설정과 그 해결(예: Vercel GitHub App, 키 동기화, Supabase 프로젝트 한도, CLI 토큰), 우회한 방법 |
| 구현 | orange-start / phase-build | **완성 시 1회** — 만든 화면 목록 + 핵심 기술 결정 2~3개 + 구현 중 막혔다 푼 지점(증상·원인·해결) |
| 이어가기 | orange-resume | 어디서 왜 멈췄고, 돌아와 무엇부터 어떻게 이었는지 |
| 보안 점검 | orange-secure | 점검한 항목 / 결과 / 수정한 것 / 남은 위험과 그 판단 근거 |

## 좋은 항목 vs 빈약한 항목 (연결 단계 예시)

**빈약함 (이렇게 쓰지 않는다):**

```markdown
### 2026-05-30 연결
- 정한 것: Next.js + Supabase + Vercel 연결.
- 막힌 점: Supabase 로그인 문제 있었는데 해결함.
```

**풍부함 (이렇게 쓴다):**

```markdown
### 2026-05-30 연결
- **한 것**: create-next-app으로 Next.js 스캐폴드를 잡고 GitHub 레포 생성 → Vercel 연결 →
  Supabase 무료 프로젝트를 붙였다. `git push` → 자동배포 루프까지 배선 완료.
- **왜 이 순서**: Supabase를 Vercel 통합으로 붙이려면 Vercel 프로젝트가 먼저 있어야 해서,
  배포 환경을 구현보다 먼저 잡았다. (라이브 URL 확인은 화면 다 만든 뒤로 미룸.)
- **어떻게**: 키는 `vercel env pull` 대신 `supabase projects api-keys`로 직접 가져왔다
  (env pull은 값이 비어 오는 버그가 있었음).
- **막힌 점 / 바꾼 점**: ① Supabase 무료 프로젝트가 이미 2개라 새로 못 만들었다 →
  새 계정으로 프로젝트를 만들어 해결. ② `supabase login`이 토큰을 저장 못 해 계속
  미인증으로 떨어졌다 → 브라우저에서 액세스 토큰을 직접 발급해 `SUPABASE_ACCESS_TOKEN`
  env var로 넘겨 우회했다.
- **배운 것**: Supabase 무료 계정당 프로젝트 2개 한도가 있다. 수업 전에 안 쓰는 프로젝트는
  지워 두면 좋다.
```

오른쪽처럼, 나중에 같은 데서 막힌 사람이 **그대로 따라할 수 있을 만큼** 구체적으로 적는다.
