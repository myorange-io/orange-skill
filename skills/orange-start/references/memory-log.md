# 만들기 기록 — MEMORY.md 작성 규칙

프로젝트 루트의 `MEMORY.md`는 **이 앱을 어떤 과정으로 만들었는지** 남기는 기록이다.
나중에 복기하거나, 수업에서 "이렇게 만들어졌다"를 사례로 보여줄 때 쓴다.

세 스킬(orange-start · orange-resume · orange-secure)이 이 한 파일을 공유한다.

## 핵심 원칙

- **토큰을 아낀다.** 단계당 **짧은 항목 하나**(약 5~8줄)만 남긴다. 화면마다, 턴마다 길게
  쓰지 않는다 — 이 플러그인의 1번 가치는 사용량 절약이다.
- **append만 한다.** 기존 내용을 **덮어쓰지 않는다.** 늘 파일 끝에 새 항목을 덧붙인다.
  파일이 길어지는 건 정상이다 (그게 기록의 목적이다).
- **결과가 아니라 과정을 남긴다.** "무엇을 만들었다"가 아니라 **무엇을 / 왜 그렇게 정했나 /
  뭐가 막혔거나 바꿨나**를 적는다. 결정과 시행착오가 교육 사례의 핵심이다.
- **사람 말로 쓴다.** 입문자가 나중에 읽어도 이해되게, 기존 스킬과 같은 한국어 문체로.
- **날짜는 셸로 얻는다.** `date +%Y-%m-%d` 결과를 항목 제목에 넣는다.
- **별도 커밋을 만들지 않는다.** 각 단계가 어차피 하는 git commit에 `MEMORY.md`를 함께 담는다.

## 파일이 없으면 — 헤더부터 만든다 (기획 단계에서 1회)

`MEMORY.md`가 없으면 아래 헤더 + 첫 '기획' 항목으로 새로 만든다:

```markdown
# 🍊 만들기 기록 — [앱 이름]

> 이 앱을 어떤 과정으로 만들었는지 남기는 기록입니다.
> 나중에 복기하거나, 다른 사람에게 사례로 보여줄 때 씁니다.

## 한눈에 보기
- 무엇을: [PLAN.md 한 줄 소개]   · 누구를 위해: [주 사용자 / 보여줄 사람]
- 시작: [YYYY-MM-DD]   · 라이브: 배포 전

---

## 기록

### [YYYY-MM-DD] 기획
- 정한 것: [핵심 흐름·화면 수·로그인/LLM 여부 등 한두 줄]
- 왜: [그렇게 정한 이유 — 현재 방식의 불편, 보여줄 사람 기준 등]
- 막힌 점 / 바꾼 점: [범위를 줄였거나 방향을 튼 지점. 없으면 "없음"]
```

이미 `MEMORY.md`가 있으면 **헤더를 다시 만들지 말고** 아래 항목 형식만 파일 끝에 덧붙인다.

## 항목(entry) 형식 — 매 단계 공통

```markdown
### [YYYY-MM-DD] [단계 이름]
- 정한 것: …
- 왜: …
- 막힌 점 / 바꾼 점: …
```

단계 이름은 그 단계에 맞게 쓴다: `기획` · `연결` · `구현` · `이어가기` · `보안 점검`.
배포 URL이 처음 생기면 '한눈에 보기'의 `라이브:` 한 줄만 실제 URL로 갱신한다 (이건 덮어쓰기 예외).

## 단계별 무엇을 적나 (요약)

| 단계 | 스킬 | 남길 것 |
|------|------|---------|
| 기획 | orange-start / phase-plan | 헤더 생성 + 무엇을·누구를·왜 이 범위로 정했나 |
| 연결 | orange-start / phase-connect | 스택·배포·DB를 어떻게 붙였나, 막힌 설정이 있었나 |
| 구현 | orange-start / phase-build | **완성 시 1회** — 만든 화면 목록 + 핵심 결정 1~2개 |
| 이어가기 | orange-resume | (선택) 어디서 멈췄고 무엇부터 잇는지 한 줄 |
| 보안 점검 | orange-secure | 점검 결과 / 수정한 항목 / 남은 위험 |
