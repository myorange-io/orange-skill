# 구현 단계 — 화면을 하나씩 만든다

목표: `PLAN.md`의 화면을 **한 번에 하나씩** 진짜로 만든다. 화면 하나가 끝날 때마다 저장·배포되고,
사용자가 직접 확인한다.

## 준비

1. `PLAN.md`를 읽는다 — 특히 `## 진행 상황` 체크리스트와 `## 화면`·`## 데이터`.
2. `design/` 폴더의 Stitch 산출물을 시각 참고용으로 둔다 (있다면).
3. 개발 서버를 켠다: `npm run dev` (백그라운드 실행). 사용자에게 http://localhost:3000 을
   열어두라고 안내한다. 코드를 저장하면 화면이 자동으로 새로고침된다.

## 세로 슬라이스 루프

진행 체크리스트에서 **아직 안 끝난 '구현' 항목 하나**를 고른다. 그 화면 하나를 끝까지 만든다:

### 1) 데이터 준비
이 화면이 쓸 Supabase 테이블이 아직 없으면 만든다.
- 필요한 테이블의 SQL(`create table ...`)을 작성해 `supabase/` 폴더에 `.sql` 파일로 저장한다
  (예: `supabase/001_<테이블>.sql`). 사용자에게도 그 SQL을 보여준다.
- **CLI로 바로 실행한다** (연결 단계에서 `supabase link`를 해 둔 상태):

  ```bash
  npx supabase db query --linked --file supabase/001_<테이블>.sql
  ```

  `db query --linked`는 Management API로 실행되어 DB 비밀번호가 필요 없다. 오류가 나면 같은
  폴더의 `troubleshooting.md` 'Supabase 테이블 생성 실패' 항목을 본다.
- 로그인이 필요 없는 앱이면 RLS는 기본값(꺼짐)으로 둔다 — anon 키로 읽고 쓸 수 있다.
  로그인이 있는 앱이면 Supabase Auth(이메일/비밀번호)를 쓰고, 테이블에 `user_id` 컬럼과
  `auth.uid() = user_id` RLS 정책을 둔다.

### 2) 화면 구현
- `app/` 아래에 해당 화면의 페이지(`app/<경로>/page.tsx`)와 필요한 컴포넌트를 만든다.
- `design/`의 해당 화면을 참고하되 **그대로 복사하지 않는다** — React 컴포넌트와 Tailwind로
  다시 구현한다. Stitch HTML은 출발점일 뿐이다.
- **스타일은 Tailwind 유틸리티 클래스(`className`)로** 한다. 이 프로젝트는 Tailwind v4라 CSS
  파서가 엄격하다 — `app/globals.css`는 되도록 손대지 않는다. 첫 줄 `@import "tailwindcss";`는
  지우지 말고, 그 **뒤에 다른 `@import`(웹폰트 등)를 넣지 않는다** — 빌드가 깨진다. 웹폰트가
  필요하면 `app/layout.tsx`에서 `next/font/google`을 쓴다 (한국어는 시스템 폰트로도 충분하다).
- `lib/supabase.ts`의 `supabase` 클라이언트로 데이터를 읽고 쓴다.
- **URL·식별자는 ASCII만 쓴다.** 한글은 화면 표시용으로만 — URL 경로·slug·DB 조회 키에 한글이
  들어가면 URL 인코딩이 꼬여 조회가 깨진다(Supabase 406 등). 한글 이름에서 slug를 만들 땐
  한글을 남기지 말고 짧은 랜덤 ASCII id를 쓴다 (예: `crypto.randomUUID().slice(0, 8)`).

### 3) 사용자 확인
- localhost:3000의 해당 화면을 사용자가 직접 보게 한다: "이 화면 어때요? 생각대로 됐나요?"
- 수정을 요청하면 반영하고 다시 보게 한다. (이 확인은 Claude 사용량을 쓰지 않는다.)

### 4) 저장 + 배포
화면이 동작하면:

```bash
git add -A && git commit -m "구현: <화면 이름>"
git push
```

`git push`하면 Vercel이 자동 배포한다. (자동 배포가 안 되면 `vercel --prod`.)
`PLAN.md` 진행 체크리스트에서 이 화면 항목을 `- [x]`로 바꾼다.

`✅ <화면 이름> 완료 — 저장·배포됨. 멈춰도 됩니다.` 를 출력한다.

### 다음 화면으로
체크리스트에 안 끝난 화면이 남았으면 1)로 돌아간다. **한 번에 한 화면씩.**

## 막혔을 때

오류가 나면 추측으로 이것저것 바꾸지 않는다:
1. **오류 메시지를 그대로 읽는다.** 어디서, 무엇이 문제인지 정확히 본다.
2. 같은 폴더의 `troubleshooting.md`를 읽어 흔한 원인과 맞는지 본다.
3. **근본 원인**을 고친다. 증상만 가리지 않는다.
4. 다시 확인한다.

## 마무리

모든 화면 항목이 `- [x]`가 되면:

1. `npm run build`로 배포용 빌드가 통과하는지 확인한다. 실패하면 고친다.
2. 라이브 URL을 열어 마지막으로 확인한다.
3. `README.md`에 이 앱이 무엇인지 한 단락 적는다.
4. `PLAN.md`의 `- [ ] 배포 확인`을 `- [x]`로 바꾸고 커밋·푸시한다.
5. 사용자에게 **공유 가능한 라이브 URL**을 보여주며 축하한다. (발표·데모에 바로 쓸 수 있다.)

## 원칙

- **스코프 준수** — `PLAN.md`에 없는 화면·기능을 임의로 더하지 않는다. 새 아이디어는 `PLAN.md`의
  '제외 — 다음에'에 적어두고 넘어간다.
- **작은 슬라이스** — 한 번에 한 화면. 여러 화면을 동시에 건드리지 않는다.
- **단순하게** — 입문자 MVP다. 추상화·불필요한 예외처리를 만들지 않는다. 가장 단순한 코드를 쓴다.
- **사용자가 확인** — 화면 검증은 사용자가 브라우저로 한다. 자동 브라우저 검증·서브에이전트는 쓰지 않는다.
