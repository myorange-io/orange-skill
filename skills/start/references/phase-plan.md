# 기획 단계 — 적응형 인터뷰

목표: 사용자의 아이디어를 **2~3번의 짧은 질문 라운드**로 다듬어, 한 페이지짜리 `PLAN.md`와
Stitch용 프롬프트를 만든다. 질문이 앞 답변에서 이어지되 왕복은 최대 3번이다.

## 시작: 아이디어 파악

1. 사용자의 아이디어를 듣는다 (이미 적었으면 그대로 사용). 비어 있으면 "어떤 걸 만들고
   싶어요?" 라고 한 번만 묻는다.
2. 아이디어를 8가지 유형 중 하나로 마음속에 분류한다 (사용자에게 말할 필요 없음):
   업무 자동화 · 내부 도구/대시보드 · 웹앱/신청 페이지 · 데이터 정리/분석 ·
   문서/메일/제안서 자동화 · 챗봇/FAQ · 운영 프로세스 개선 · 발표용 MVP/데모.
   → 어떤 유형이든 **Next.js 웹앱 하나로** 표현한다.
3. 아이디어 설명에서 **이미 답이 나온 항목은 메모**해 둔다. 그건 다시 묻지 않는다.

## 인터뷰 규칙

- **AskUserQuestion에는 사용자에게 보여줄 완성된 질문과 선택지만 넣는다.** 이 파일의 안내
  문구는 *너에게 주는 지시*다. '어떤 질문을 고를지', '질문 가이드', '아래에서' 같은 네 작업
  과정을 질문·선택지 텍스트에 절대 넣지 않는다. 질문은 자연스럽고 완전한 한국어 문장으로 쓴다.
- **한 라운드 = AskUserQuestion 한 번.** 질문 3~4개를 묶어서 던진다.
- 각 질문에는 아이디어에서 추론한 **구체적 선택지 2~4개**를 만들어 준다. 입문자가 글을 쓰는
  대신 고르게 한다 ("기타"로 직접 입력도 가능하다).
- **Round 2·3의 질문은 직전 답변에서 생성한다.** 고정 질문지가 아니다.
- 이미 아는 항목은 건너뛴다. 질문 문구는 짧게 — 긴 설명을 붙이지 않는다.
- **라운드 사이마다 지금까지의 `PLAN.md` 초안을 대화창에 보여준다** — 기획이 채워지는 게
  눈에 보이게. 완성된 `PLAN.md` 파일은 인터뷰가 끝난 뒤 한 번 쓴다.

## Round 1 — 무엇 · 누구 · 핵심

사용자에게 아래 질문들을 **한 번의 AskUserQuestion**으로 묻는다. 아이디어 설명에서 이미 답이
나온 질문은 뺀다 (보통 3~4개를 묻게 된다). 각 질문의 선택지는 사용자 아이디어에서 추론해
2~4개씩 만든다 — 사용자는 "기타"로 직접 입력할 수도 있다.

물어볼 질문 — 아래 문장을 쓰되 '이 도구' 자리에 사용자 아이디어의 실제 대상을 넣어 자연스럽게 만든다:

1. 이 도구를 한 줄로 설명하면 무엇인가요?
2. 이 도구를 주로 누가 쓰나요?
3. 사용자가 이걸로 하는 가장 중요한 행동 하나는 무엇인가요?
4. 잘 되면 사용자에게 어떤 좋은 순간이 생기나요?

답을 받으면 `PLAN.md`의 **1·2절(한 줄 소개·핵심 흐름) 초안**을 대화로 보여준다:
"여기까지 이렇게 이해했어요 — 맞나요?"

## Round 2 — 화면 · 데이터

Round 1의 핵심 흐름을 바탕으로:

1. **화면 목록을 대화창에 글로 적어 보여준다.** 핵심 흐름을 3~5개 화면으로 쪼개고, 각 화면마다
   **무엇이 보이고 사용자가 무엇을 하는지** 한 줄로 적는다. 화면 이름만 나열하면 사용자가
   판단할 수 없다 — 반드시 내용을 적는다. 예:

   ```
   제안하는 화면 구성:
   ① 신청 폼 — 이름·연락처·신청 내용 입력칸과 '제출' 버튼
   ② 신청 목록 — 들어온 신청을 표로 표시, 한 줄을 누르면 상세로 이동
   ③ 상세 보기 — 신청 하나의 전체 내용 + '처리 완료' 버튼
   ```

   이어서 알린다: **실제 화면 디자인은 다음 Stitch 단계에서 눈으로 보고 고친다 — 지금은
   "어떤 화면이 필요한지"만 정하는 단계다.** 질문은 이 목록을 적은 뒤에 던진다.
2. 그다음 한 번의 AskUserQuestion으로 묻는다 (3~4개):
   - **화면 목록** — "위에 적은 화면들로 충분할까요?" → "그대로 좋음 / 화면 추가 / 화면 빼기 / 둘 합치기".
   - **데이터** — "어떤 정보를 저장해야 하나요?" → 추론한 데이터 항목들을 선택지로 (복수 선택).
   - **로그인** — "로그인이 필요한가요?" → "필요 없음(누구나) / 필요함(사람별 데이터) / 관리자만".
   - **외부 연동** (필요해 보일 때만) — 메일 발송·AI 응답·파일 업로드 등이 필요한지.

답을 받으면 `PLAN.md`의 **3·4절(화면·데이터 모델)까지 초안**을 대화로 다시 보여준다.

## Round 3 — 범위 다듬기 (조건부)

다음 중 하나라도 해당하면 Round 3을 진행한다:

- 화면이 **6개를 넘는다.**
- 한 세션(몇 시간) 안에 다 만들기 **벅차 보인다.**
- 핵심 흐름이 **여전히 모호하다.**

진행한다면 한 번의 AskUserQuestion으로:

- "한 번에 다 만들기엔 양이 많아요. **이번 MVP에 꼭 필요한 화면**을 골라주세요." (복수 선택,
  선택지 = 화면 목록.) 고르지 않은 화면은 `PLAN.md`의 '제외 — 다음에'로 보낸다.

해당 사항이 없으면 Round 3을 건너뛴다. 대신 네가 직접 MVP 범위를 좁혀 `PLAN.md`에 적은 뒤
"이렇게 범위를 잡았어요"라고 보여주고 수정받는다.

## PLAN.md 작성

인터뷰가 끝나면 프로젝트 루트에 `PLAN.md`를 아래 형식으로 쓴다. 한 페이지를 넘기지 않는다.
진행 체크리스트의 '구현' 항목은 화면 목록에 맞춰 하나씩 만든다. (git 커밋은 다음 '연결'
단계에서 시작하므로 여기서는 파일만 저장하면 된다.)

```markdown
# [프로젝트 이름]

## 한 줄 소개
[무엇을 / 누구를 위해]

## 핵심 흐름
[사용자가 하는 단 하나의 핵심 동작 — 시작부터 '성공의 순간'까지]

## 화면
1. [화면 이름] — [무엇을 하는 화면]
2. ...

## 데이터 (Supabase 테이블)
- **[테이블명]**: [컬럼], [컬럼], ...

## 기술 스택
Next.js (App Router, TypeScript) · Tailwind CSS · Supabase · Vercel

## MVP 범위
- 포함: [이번에 만들 것]
- 제외 — 다음에: [나중으로 미룰 것]

## 진행 상황
- [x] 기획 완료
- [ ] Stitch 프로토타입
- [ ] 연결 (GitHub · Vercel · Supabase)
- [ ] 구현: [화면 1 이름]
- [ ] 구현: [화면 2 이름]
- [ ] 배포 확인
```

## Stitch 프로토타입으로 넘기기

화면을 코드로 만들기 전에, 사용자가 **눈으로 먼저 확인**하게 한다.

1. 확정된 화면 목록으로 **Stitch 프롬프트**를 만든다. 영어로 간결하게, 화면별 UI 요소를 적고
   UI 텍스트는 한국어로 지정한다. 예:

   ```
   Create a web app with these screens:
   1. [Screen]: [layout, key UI elements]
   2. ...
   Style: clean, modern, friendly. Korean language UI labels.
   ```

2. 사용자에게 안내한다:
   - https://stitch.withgoogle.com 에 접속해 위 프롬프트를 붙여넣는다.
   - 화면을 생성하고, 마음에 들 때까지 다듬는다.
   - 다 됐으면 Stitch의 **'프로젝트 다운로드'**로 zip 파일을 받는다.
   - 받은 **zip 파일을 이 프로젝트 폴더 안에 그대로 넣어두고**, 돌아와서 알려준다. 압축은
     풀지 않아도 된다 — 다음 '연결' 단계에서 Claude가 풀어 `design/` 폴더로 정리한다.
     (zip을 받기 어려우면 각 화면 스크린샷을 찍어 `design/` 폴더에 넣어도 된다.)

3. 다음을 꼭 알린다:
   - **Stitch 작업은 Claude 사용량을 쓰지 않는다.** 충분히 시간을 들여도 된다.
   - **Stitch 결과가 생각과 다르면, 그건 기획이 어긋났다는 신호다.** `PLAN.md`를 고치고
     프롬프트를 다시 만들어 Stitch를 다시 돌린다.

`✅ 기획 완료 — PLAN.md 저장됨. 멈춰도 됩니다.` 를 출력한다. 사용자가 Stitch에서 돌아오면
연결 단계(같은 폴더의 `phase-connect.md`)로 잇는다.
